# Wave 2 — Unified Agent + State Refactor

> **Status:** Not started.
> **Estimated effort:** ~3 weeks. This is the biggest piece of work in the overhaul.
> **Prerequisite:** Wave 1 Track B must be done — we need the latency baseline before optimizing further. (Wave 1 Track B also delivers `usageCheckedMutation` in `/packages/db/src/convex_helpers.ts`, but that wrapper is narrowly scoped to `sendMessage` / `create` and pre-fetches `userPlan` / `userInfo` / `usage` only — it is NOT the batching layer Wave 2 needs. Wave 2's batching layer is a separate concern built inside the agent action code path; see Phase 2b.)
> **Wave 1 Track A is not a hard prerequisite, but doing it first makes the UI consumers easier to migrate as part of Phase 2c.**

---

## What this wave merges

This wave intentionally fuses **three** of the original six initiatives into a single phased refactor, because they all rewrite the same files:

1. **"Remove thread state from db"** (https://www.notion.so/3433b59a4f9c807ea686d38ff2fcc660) — move thread state to a `threadEvents` table, derive state from events.
2. **"Tighter integration of agent code"** (https://www.notion.so/3433b59a4f9c806aa606f681e59ebe46) — collapse the per-step / per-delta `ctx.runMutation(component.*)` hops left over from the original Convex Component architecture.
3. **The mutation-batching part of "Improve mutation and query speed"** (https://www.notion.so/33f3b59a4f9c80c2a48fcab3aa8f1ee5) — the prefetched-context piece is in Wave 1; the batching piece (write-through buffer for delta/message saves) belongs here, layered on top of the extended context wrapper.

### Why merged (recap from planning conversation)

The agent-integration subagent counted **5+ mutation hops per stream delta** via `ctx.runMutation(component.streams.*)` and a **3-step `setState` dance per generation** in `agents.ts`. Both the events table refactor (#1) and the agent-hops refactor (#4) need to rewrite `streamResponse` and `delta_streamer.ts`. Doing them serially means rewriting the same code twice. The mutation-batching slice (#3-batching) provides the mechanism that makes both refactors cleaner.

## Source-of-intent quotes

### From "Remove thread state from db"

> There are a lot of different states a thread can be in while a new response is being generated by an LLM. Currently what we're doing to handle this is storing a state variable in the thread table that shows if it's idle, if the LLM is streaming our response, and stuff like that.
>
> Generally though, this is not how you want to use a database to be pretty broad. Databases aren't good at storing state, they're good at storing data. So what I want to do is move to a system where instead of storing state in the database, we are solely storing transactions or events rather.
>
> I think the way to do this is to introduce a new table called Thread Events, perhaps. And in this table, we store things that happen. So when a user sends a new message, we store an event called like New User Message, perhaps. And then when the LLM starts streaming a message back, we store a new entry in the events table that says "Hey, message is being shared back" and then when it completes we store our new event that says "Response completed" If the user tries to abort, perhaps we store a new message that says "Triggered Message Abortion"
>
> this way, we are just storing events in the database. And then when we need to derive state in the web apps, we derive the state from the data instead of just pulling the state directly from the database. I think this will make it less likely that we get stuck in weird states where threads are generating forever and it'll make it a little easier to reason about how things work.

### From "Tighter integration of agent code"

> Originally, a lot of the code for generating responses from agents lived in convex's agent package, which was a convex component. But since my project is so heavily centered around the use of this component, I figured it would make more sense to integrate it directly into my code base so that I could have a tighter integration between what I write and what the agent code runs.
>
> This is working fine right now, but I have a feeling we can tighten the integration and improve performance and general readability and maintainability of the code.
>
> For example, I'm pretty sure that right now we do quite a few different hops during the agent generation due to the fact that the code was originally intended to run in, convex components run in an isolated sort of environment. So we had to do some hops between the components environments, but now that it's all in one environment, which is my main app, we should be able to reduce the the number of hops and whatnot so that the code executes faster. So for example, like hopping over to node actions, hopping between actions, running mutations or queries from inside an action, like all of these add overhead and as many of them as we can remove and you know tighten down would be good.

## Current state findings (from research subagents)

### State storage today

- **Schema:** `/packages/db/src/agent/schema.ts` (around lines 22–27). The `threads` table has a `state` field (`"idle" | "waiting" | "streaming"`) plus `generationFnId` for tracking the scheduled function.
- **Adjacent state lives in:**
  - `streamingMessages` table — has its own `state: { kind: "streaming" | "finished" | "aborted" }`.
  - `messages` table — has `status: "pending" | "success" | "failed"`.
- **Mutations that flip thread state:** `/packages/db/src/ai/thread/mutations.ts` — `setState`, `sendMessage` (→ `"waiting"`), `abortGeneration` (→ `"idle"`), `resetIdleIfStreaming`, `create` (→ `"waiting"`).
- **The state machine inside `streamResponse`:** `/packages/db/src/ai/agents.ts` (around lines 73–183) — calls `setState("streaming")` after init, then `resetIdleIfStreaming` on completion. Has guard logic and code comments around lines 148–150 explaining race conditions: "set thread back to idle — only if we're still the active stream; a user-triggered abort (or a subsequent generation) will have already moved the thread out of the 'streaming' state".
- **Stuck-state protections that exist today** (these are the bugs the user wants to make impossible):
  - `wasAborted` query (around lines 251–261).
  - `resetIdleIfStreaming` (around lines 263–274) — only resets if currently `"streaming"`.
  - System error codes G1, G2 logged when stream init or streaming fails, then `setState("idle")`.
- **UI consumers of state** (after Wave 1 Track A, `useThreadStatus` is deleted; all reads go through `useQuery(threadQueries.state(threadId))` at the call site, where `threadQueries` lives in `/packages/features/src/thread/lib/queries.ts`):
  - `/apps/web/src/features/thread/thread.tsx` — derives `isThreadIdle` from the query result.
  - `/apps/web/src/features/thread/components/thread-list-item.tsx` — pulses brain icon when `state === "streaming" || state === "waiting"`.
  - `/packages/features/src/composer/hooks/use-send-message.ts` — derives its own `isThreadIdle`.
  - `/apps/web/src/features/composer/components/composer-send.tsx` — derives `isGenerating`.
  - `/packages/features/src/thread/types/thread-types.ts` — exposes the type to the client.

### Agent hops today

- **Agent code root:** `/packages/db/src/agent/` — ~78 files, ~9,400 LOC.
- **The component shim:** `/packages/db/src/agent/component.ts` (around line 5):
  > `export const agentComponent: AgentComponent = internal.agent as any;`
  All agent calls go through `ctx.runMutation/runQuery/runAction(agentComponent.*)` even though it now lives in the same Convex environment.
- **Hottest hop sites:**
  - `/packages/db/src/agent/client/start/save.ts` (around lines 38–40) — `ctx.runMutation(component.messages.addMessages)` per step.
  - `/packages/db/src/agent/client/delta_streamer.ts` (around lines 148, 161, 222, 274, 288) — **5 separate stream-related mutations per stream lifecycle.**
  - `/packages/db/src/agent/client/start.ts` (around line 211) — `component.messages.finalizeMessage`.
  - `/packages/db/src/agent/client/index.ts` (around line 170) — `component.threads.updateThread`.
  - `/packages/db/src/ai/agents.ts` (around lines 104–118, 125, 151, 157, 173–175) — 5× `runMutation/runQuery` for state + follow-ups.
- **Intrinsic hops (do NOT remove):**
  - Node action → mutation boundaries forced by node-only deps.
  - External API calls (image gen in `tools/image/actions.ts`).
  - LLM provider round-trips.
- **Accidental hops (remove):**
  - All `component.messages.*`, `component.streams.*`, `component.threads.*` calls that route through `internal.agent` despite the function living in the same environment.
- **Mapping layer** at `/packages/db/src/agent/mapping/` (8 files) — handles message format conversion. Touch carefully; serialization boundaries are deep here.

## Phase 2a — Events table as additive audit log (~1 week, very low risk)

### Goal

Land the `threadEvents` table as a **write-only audit trail**. Nothing reads it yet. Pure dual-write so we can validate the event taxonomy in production with zero risk to behavior.

### Steps

1. **Design the event taxonomy.** Recommended starting set (confirm with user — see Q1):
   - `user_message_sent`
   - `generation_scheduled`
   - `streaming_started`
   - `streaming_completed`
   - `streaming_aborted`
   - `generation_failed` (with error code metadata, e.g. G1/G2)
   - `thread_created`
2. **Schema change** in `/packages/db/src/agent/schema.ts`:
   ```
   threadEvents: defineTable({
     userId: string,
     threadId: Id<"threads">,
     timestamp: number,
     eventType: union of literals above,
     metadata: optional(any),       // e.g. { errorCode: "G1", reason: "user-aborted" }
     generationId: optional(string), // groups events from one generation cycle
   }).index("threadId_timestamp", ["threadId", "timestamp"])
   ```
3. **Emit events from existing call sites** without removing any existing `setState` calls (dual-write):
   - `sendMessage` mutation → `user_message_sent`, `generation_scheduled`.
   - `create` mutation → `thread_created`, `user_message_sent`, `generation_scheduled`.
   - `streamResponse` action → `streaming_started`, `streaming_completed` or `generation_failed`.
   - `abortGeneration` mutation → `streaming_aborted`.
4. **Soak in production for at least a few days.** Manually inspect the events table; verify the taxonomy captures all real-world cases. **Do not proceed to Phase 2b/2c until confident.**

### Open questions

- **Q1.** Confirm the starting event taxonomy. Are these the right names? Are there events missing (e.g., `tool_call_started`, `tool_call_completed`, `step_completed`)? The Notion doc lists "New User Message", "Streaming Started", "Response Completed", "Triggered Message Abortion" as examples — confirm these are exhaustive enough.
- **Q2.** Should events carry a `generationId` (UUID grouping events from one generation cycle)? Recommended yes — makes derivation trivial. Confirm.
- **Q3.** Retention: do events live forever, or do we plan to prune? (Probably forever for now; ask anyway.)
- **Q4.** Do XR / mobile apps need to also write events from the client side, or are events strictly a backend concern derived from server actions? Recommended: backend-only writes.

### Done criteria

- New table exists, indexed.
- Events written from all 4 mutation/action sites listed above.
- Manual inspection confirms taxonomy is correct against real generation cycles, including aborts and errors.
- Zero behavior change visible to users.

## Phase 2b — Agent pipeline collapse (~1.5 weeks, medium risk)

### Goal

Reduce the per-generation mutation count dramatically by:
1. Batching `streams.create / addDelta / finish` calls into a buffered mutation per stream lifecycle.
2. Batching `messages.addMessages` saves at step boundaries.
3. Replacing `ctx.runMutation(component.X)` indirections with direct function calls where possible.

This is the part that addresses the user's Notion intent: "we should be able to reduce the the number of hops and whatnot so that the code executes faster."

### Steps

1. **Design the batching layer.** This is a separate concern from Wave 1 Track B's `usageCheckedMutation` — that wrapper pre-fetches user/plan/usage for the two entry-point mutations; batching happens later, inside agent *actions* (different Convex function type). The batching layer is a write-through buffer (e.g. a `StreamWriter` helper class or a context-scoped object passed through the agent pipeline) that queues mutation calls and flushes them in batches. Decide buffering granularity (per-step? per-stream? — see Q5). Do NOT try to reuse `usageCheckedMutation`'s context shape for this; they serve different purposes.
2. **Convert delta-streamer writes** in `/packages/db/src/agent/client/delta_streamer.ts` (lines ~148, 161, 222, 274, 288) to batched writes flushed at stream-end (or at periodic intervals if the stream is long — the deltas need to land in the DB visibly during streaming for the UI; balance latency vs. write count — see Q6).
3. **Convert message saves** in `/packages/db/src/agent/client/start/save.ts` (lines ~38–40) to batch per step rather than per message.
4. **Inline the component shim** where the function is in the same environment. Replace `ctx.runMutation(agentComponent.threads.updateThread, args)` with a direct import of the function from `/packages/db/src/agent/handlers/threads.ts` (or wherever it lives) and call it directly. Same for `messages` and `streams` handlers.
5. **Audit `agents.ts`** (lines ~104–118, 125, 151, 157, 173–175) for `runMutation/runQuery` calls that can become direct calls now that the agent code is in-app.
6. **Re-run the instrumentation from Wave 1 Track B** for `sendMessage` end-to-end (mutation → action → first delta visible to client). Report numbers.

### Open questions

- **Q5.** Batching granularity for stream deltas. Two extremes: (a) write every delta immediately (current behavior, 5+ writes per stream) vs. (b) buffer all deltas and write at stream end (1 write but UI sees nothing until stream ends). The reasonable middle: write every N deltas or every M ms. **What latency is acceptable for the streaming UI? 50ms? 100ms?**
- **Q6.** During streaming, the UI subscribes to `streamingMessages` and renders deltas as they arrive. If we batch deltas, we trade DB-write count for UI smoothness. **Confirm this tradeoff is acceptable**, or insist on per-delta writes and only batch other things.
- **Q7.** Are there places where `ctx.runMutation(component.X)` is intentional because of transactional boundaries (e.g., a mutation needs its own transaction)? If yes, we cannot inline those. Need to audit case-by-case.
- **Q8.** Approval flows insert extra tool-results mid-stream (per a comment in `save.ts` around lines 88–89). Confirm with user how the batching should interact with these mid-stream inserts. May need a flush-before-insert.
- **Q9.** Does the user want this phase to also remove the `component.ts` shim entirely, or keep it as a thin alias for backwards-compat in case they ever extract the agent back into a Convex Component?

### Validation

- `pnpm run lint`, `pnpm run typecheck`, `pnpm run format:fix`.
- Streaming generation works end-to-end: open a thread, send a message, watch deltas arrive, abort midway, retry — all observed behavior matches pre-change.
- Tool calls work (image gen, search, etc. — whatever tools exist).
- Latency improvement vs. baseline: report numbers in PR description.
- No new errors in production logs over a soak period.

### Done criteria

- Stream lifecycle uses ≤ 2 mutations (down from 5+).
- Per-step message save uses 1 mutation per step (down from 1 per message).
- All accidental `ctx.runMutation(component.X)` hops are either inlined or have a written justification for staying.

## Phase 2c — Switch reads to derived state, drop `state` field (~0.5 weeks)

### Goal

Make events the source of truth. Stop reading `thread.state`. Drop the field.

### Steps

1. **Add `getStateFromEvents` query** in `/packages/db/src/ai/thread/queries.ts`. Implementation strategies:
   - Naive: query latest event for thread, derive state from event type. O(1) read with the `threadId_timestamp` index.
   - Cached: denormalize a `lastEventType` + `lastEventAt` on the `threads` row, validate against recent events. Required if we worry about long event lists slowing things down (subagent flagged this — see Q10).
2. **Feature-flag the switch** inside the backend `getState` query (which `threadQueries.state(threadId)` points at) so we can flip back if the new derivation has a bug.
3. **No hook to update** — Wave 1 Track A already deleted `useThreadStatus`. Call sites use `useQuery(threadQueries.state(threadId))` directly and will automatically see the derived-from-events value once the backend query is switched. The only client-side change in this step is verifying the `threadQueries.state(...)` shape (defined in `/packages/features/src/thread/lib/queries.ts`) still matches what call sites expect after the derivation change — if the event-derived state returns the same `"idle" | "waiting" | "streaming"` union, no client edits are needed.
4. **Soak for a week** in production with the flag on. Watch for any "stuck thread" reports — if they go to zero, the events derivation is doing its job.
5. **Remove the dual-write.** Stop calling `setState` from mutations and `streamResponse`. State is now purely derived.
6. **Drop the `state` field** from the `threads` schema. Drop `setState`, `resetIdleIfStreaming`, `wasAborted` queries (or rewrite `wasAborted` to derive from events).
7. **Migrate UI consumers** (already done as part of Wave 1 Track A pilot for `useThreadStatus`; the rest of the consumers should already use derived state via the hook).

### Open questions

- **Q10.** Long-thread perf: if a thread has thousands of events, deriving state by querying the latest event should still be O(1) with the right index. But if we need to derive *full* historical state, we might need pagination. **Confirm the only thing we need to derive is "current state", not full history.**
- **Q11.** Data migration for existing threads. Two options: (a) backfill events from existing `messages` history (subagent estimated 2–3 hours for the script), (b) accept that existing threads won't have events before this date and derive their state lazily from current `thread.state` on first read until the next generation. **Pick one.**
- **Q12.** What happens to `wasAborted`? Rewrite to "is there a `streaming_aborted` event after the last `generation_scheduled`?" Confirm.
- **Q13.** Should we keep the `streamingMessages.state` and `messages.status` fields, or do those also move to events? Recommend keeping for now — they have different semantics from thread-level state. Confirm.

### Validation

- All "stuck thread" recovery paths (abort, error, timeout) work end-to-end and resolve to a sensible derived state.
- Backfill (if chosen) runs cleanly on a snapshot of production data.
- `pnpm run lint`, `pnpm run typecheck`, `pnpm run format:fix`.

### Done criteria

- `thread.state` is gone from schema.
- All UI reads of state go through the new derived query.
- No "stuck streaming" bugs reported during the soak period.

## Risks specific to this wave

- **Schema migration in a live system.** Phase 2c drops a field. Coordinate carefully.
- **Race conditions in stream abort.** The current code has guards (`resetIdleIfStreaming` only resets if state is `"streaming"`). With events, multiple `streaming_completed` events should be idempotent, but the abort path needs to be tested heavily.
- **Mapping layer fragility.** `/packages/db/src/agent/mapping/` has 8 files of message format conversion. Touching this in Phase 2b can break message rendering. Prefer changes that don't go through this layer.
- **Approval flows.** Mid-stream tool-result inserts (see Q8) interact with batching in subtle ways.

## Cross-wave dependencies

- **Requires** Wave 1 Track B for the baseline latency numbers (so we can measure Wave 2's impact). Does NOT depend on Track B's `usageCheckedMutation` — the batching layer in Phase 2b is a separate construct.
- **Should be done before** Wave 3 (events provide cleaner state for optimistic updates; the abort-button delay fix in Wave 3 Phase 3a relies on the `threadQueries.state` query reading from events instead of the stale `state` field).
- **Wave 4 (Effect)** can be layered on top of this wave's `streamResponse` if desired.
